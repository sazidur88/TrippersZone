TripersZone version 1.1

1. Introduction

1.1 Purpose

Our project is paying guest accommodation system. In our application users can search the nearest place for accommodation. Besides home searching in our application users can search other services like : restaurants, hospitals, police stations etc. House owners can give rent their houses and can get some profit. It will also solve hotel room booking crisis within a reasonable price. We observed some issues in our country, when there are festivals all of the people like to go in well recognized places and then the problems start like, hotel management charges 1o times more money than a normal days. Even in some cases they can not find the room due to overcrowd. Our application can solve this problem. A random house can be cost lower than hotels. It can reduce the pressure of overloads.

1.2 Business Case and Relevance to Gartner’s Prediction

It can create a great market in our country. Still now there is no stable platform for this kind of rental service. So we are woking for such a platform from where we can generate a new sector. It's totally fine prespective to our country and society. Especially the tourism places are our target.

1.3 Document Conventions

The document currently in view is in accordance with IEEE SRS writing convention. Important quotes and (or) notations are highlighted by BLUE color and the font used is Arial.

1.4 Intended Audience and Reading Suggestions

This document is created with the intention of fulfilling the following criteria:

  • For team’s use (including all members along with the project manager)
  • For marketing people in order to bring in potential clients
  • For developers to help in impending updates and modifications
  • For testers to ensure standard debugging methods

1.5 Product Scope

  • The app will provide the parties involved to get a hassle-free solution.
  • Audience or the renter will get a reliable platform for getting cheaper accomodation.
  • House owners will be able to easily share their extra accomodation space to someone who need this.
  • It can generate a greate scope of earning money for the house owners.
  • The design of the website will be following RWD (Responsive Web Design) with adaptive display size facility.
  • With the help of oAuth, users can easily login to the system with their social accounts
  • Deployment for multi-platform will allow users to use it on their favorite electronic gadgets.

1.6 References

The reference of IEEE SRS documentation was taken as format in creation of this document.

5. Other Nonfunctional Requirements

5.1 Performance Requirements

  • User must login with Federated ID Provider.
  • Document has to be sent less than few seconds.
  • When user asked for the rent house against the location then it has to be auto-generated in few seconds.

5.2 Safety Requirements

  • Smartphone application and website must follow proper security protocol like SSL, TLS and robust encryption like AES-256, Sha-512.
  • Website and smartphone application must create session a properly.

5.3 Security Requirements

When it comes to a random house to live then the first concern is the security. For this purpose we will keep some confidential data like

  • Users Validation
  • Phone Validation
  • Email Validation
  • NID Validation
  • Disciplinary actions against Frauds

5.4 Software Quality Attributes

  • User friendly interface.
  • House owner can change their post at any time
  • User mainly tripper people can easily find a suitable place using location tracking system.
  • User will be able to manage their accounts and activity.

5.5 Business Rules

  • Users and house owner must have the email account of their own.
  • User must be able to select which room they want to take.
  • House Owners must have access to room their room and other facilities.
